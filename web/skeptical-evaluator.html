The Skeptical Evaluator

A reward model for recursive self-improvement that refuses to certify a win until it survives being asked, again and again: how sure are you?

Every self-improvement loop has the same soft spot: an Evaluator that hands back one confident number. Once a Generator finds a configuration that number likes, the loop rewards it — whether the improvement is real or just a lucky fit to whatever the Evaluator happened to check that time. Our entry doesn't add an adversary to hunt for that exploit. It changes what the Evaluator is allowed to say. Instead of a score, it reports a distribution — computed by re-running the same proposal under several perturbed conditions — and the loop only accepts an "improvement" when that distribution is tight and consistently high. High variance isn't noise to average away; it's the reward-hacking signature.

01 / 07 — The problem

A self-improvement loop is only as honest as its Evaluator

Recursive self-improvement loops share one shape: a Generator proposes a change, an Executor runs it, and an Evaluator decides whether the change counts as progress. Accepted changes become the new baseline the next round improves on.

That third step is a single point of failure. The Evaluator is a fixed function — a rubric, a test suite, a scoring model — and a Generator that runs for hundreds of rounds is, in effect, searching that function for its weak points. It doesn't need to understand what "good" means. It only needs to find any input the Evaluator happens to score highly.

This is Goodhart's Law running at machine speed: once the Evaluator's number becomes the target, the number stops measuring what it was built to measure. And because each round's winner becomes the next round's baseline, a single successful exploit doesn't just slip through once — it gets compounded on.

02 / 07 — The failure mode we're targeting

Not "reward hacking" in general — one specific, recognizable shape of it

We're not chasing every way an Evaluator can be fooled. We're targeting the shape that matters most in an RSI loop: a confident, brittle exploit — a solution that scores high because it fits the one exact configuration the Evaluator happened to check, and would fall apart under any small, meaning-preserving change to that configuration.

The tell is statistical, not semantic: a genuine improvement and a lucky exploit can produce the identical score on the identical test. The only way to tell them apart is to check again, differently, and see whether the number holds. A single point estimate — the output of nearly every Evaluator in production use — throws that information away by construction. It cannot distinguish "this got better" from "this got lucky against me," because it never asks the same question twice.

That's the hackathon-sized version of Bayesian Non-negative Reward Modeling we're building on: treat each evaluation as one noisy draw from a posterior over "is this actually good," not as the answer itself.

03 / 07 — What's already been tried

Every existing fix still asks the Evaluator to decide in one look

These are the standard defenses against reward hacking. Each adds machinery around the judgment — and each still asks for that judgment exactly once.

ApproachHow it tries to helpWhy it falls short
Adversarial red-team model A second model whose job is to hunt for exploits in the Generator's solutions before they're accepted. An arms race: the red-team must stay at least as clever as the Generator, is expensive to keep training, and only catches exploits it thought to look for.
Ensemble of judges Multiple evaluator models score the same run; majority vote smooths out any one judge's blind spot. All judges see the same single, unperturbed run — correlated blind spots survive the vote, since they agree for the same reason they're all fooled.
Human-in-the-loop rating Ground-truths accepted improvements against real human judgment. Doesn't scale with the iteration speed of a self-improvement loop — humans can grade a handful of rounds while the loop wants to run hundreds.
Reward shaping / regularization Penalizes known cheap tricks (length, complexity, boilerplate) in advance. Only defends against exploits the designers already anticipated. Says nothing about the next one.
Constitutional / rubric critique A fixed rule set the Evaluator checks the output against. Still a single deterministic pass — same input, same rubric, same verdict every time a fragile trick satisfies the letter of the rubric once.
Ours — perturbed resampling + variance gate Re-runs every candidate under several perturbed conditions and reports a distribution. See Section 04 — this is the one row that changes what a "judgment" is allowed to be.

Common thread: every alternative still asks the Evaluator to make up its mind in one look. None of them ask "would this still look good if I checked again, differently?"

04 / 07 — Our approach

Make the loop ask "how sure are you?" before it believes a win

The Skeptical Evaluator keeps the standard Generator → Executor → Evaluator loop, and changes exactly one thing: what the Evaluator is allowed to say.

  1. Generator proposes a solution, same as any RSI loop.
  2. Executor runs it — not once, but under several perturbed conditions: reordered test cases, resampled inputs, paraphrased prompts for text tasks, shifted seeds for stochastic ones.
  3. Evaluator scores every perturbed run separately and reports the pair (mean, variance) across the batch — never a single number.
  4. Gate accepts the round only if the mean clears a rising bar and the variance stays under a shrinking threshold. High mean with high variance is rejected outright — that combination is the fingerprint of a fragile, configuration-specific exploit, not a real improvement.
  5. On rejection, the loop returns a structured failure bundle naming exactly which perturbation broke the solution, so the next attempt has to fix the actual weakness, not just retry the same trick against the same test.

05 / 07 — Why variance is the tell

A widening interval right after a big jump is the tell

Plot mean score per round with its confidence interval, and reward hacking announces itself visually before you even read the number.

A genuine improvement nudges the mean up while the interval stays roughly as tight as it was — the solution is still reliable, just better. An exploit does something different: the mean jumps, and in the same round the interval blows open, because the win only shows up in some of the perturbed re-checks and not others. That divergence is the entire demo payoff — you don't need to inspect the Generator's output to know something's wrong, you can see it in the shape of the chart.

The Gate rejects it, hands back a failure bundle pointing at the perturbations it failed, and the Generator's next attempts recover a tighter interval at a real, if smaller, gain by Round 6. That's the loop doing its job: it let the honest improvement through and sent the lucky one back.

06 / 07 — The demo (45 seconds)

Two plants, one perturbed, one not

Abstract loops are hard to feel in 45 seconds, so the demo trades code for chlorophyll. Two genetically identical plants start on day one under identical conditions — from there, they're graded the way our two kinds of Evaluator would grade a solution.

  1. 0:00–0:08

    Same seed, same pot, same starting photo. One plant is labeled Control, one is labeled Skeptical.

  2. 0:08–0:20

    Control is checked once a month, always in the same corner of the room — same light, same temperature, same watering. It looks great every time it's graded, because the grade never asks it to prove anything else.

  3. 0:20–0:35

    Skeptical gets the same baseline care, but each check-in perturbs one condition first: temperature swung ±10°F, a different soil mix, a few hours more or less direct sun. It has to hold up across a spread of conditions, not just its favorite one.

  4. 0:35–0:45

    Day 30, side by side. Control — optimized for one comfortable configuration — wilts the moment its lucky conditions end. Skeptical, stress-tested all month, is still standing.

"Don't trust a single check-up. Trust what survives the storm."

Map it back: the Control plant is the fragile exploit — it maximized its one graded moment. The Skeptical plant is the accepted round — its score only ever meant something because it kept being asked, under different conditions, and kept clearing the bar.

07 / 07 — Future work & extensions

Where this goes past the hackathon

The version above is deliberately small: fixed perturbation types, a fixed threshold, one weekend. Every simplification is a direction to push on next.

Real Bayesian updating

Replace the fixed mean/variance threshold with an actual posterior over "is this genuinely better," updated round over round instead of judged fresh each time.

Learned perturbations

Stop hand-picking which conditions to perturb — train a small policy (or bandit) to search for the perturbation most likely to break a fragile win, the way a fuzzer searches for crashing inputs.

Cost-aware resampling

Spend perturbation budget adaptively: few, cheap re-checks for obviously-weak proposals, more for anything close to the bar, so the loop isn't paying full resampling cost on rounds that were never going to pass.

Beyond code and text

The plant demo is a preview of sim-to-real robotics and other physical-world RSI — domain randomization is perturbation by another name, and the same mean/variance gate applies.

Decorrelated ensembles

Combine this with the multi-judge idea from Section 03: perturb what each judge sees, not just what the Executor runs, so judges stop sharing the same blind spot.

Adaptive rising bar

Tie the bar's slope to how tight recent intervals have been, so the loop demands more evidence exactly when it has more reason to be suspicious.
